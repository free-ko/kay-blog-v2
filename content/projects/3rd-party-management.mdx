# 외부 라이브러리를 관리 하는 방법

### 왜 의존성을 줄여야 할까요?

외부 라이브러리를 사용할 때는 의존성을 최소화하는 것이 중요합니다. 외부 라이브러리가 업데이트되거나 문제가 발생하면, 우리 프로젝트에도 큰 영향을 미칠 수 있기 때문입니다. 그렇다면 외부 라이브러리와의 의존성을 어떻게 하면 줄일 수 있을까요?

<br />

### 의존성 줄이는 방법

최근 [쏙쏙 들어오는 함수형 코딩 책](http://www.yes24.com/Product/Goods/108748841)을 바탕으로 진행한 [함수형 프로그래밍 스터디](https://free-ko.github.io/study-js-functionalProgramming-1/)에서, 외부 라이브러리의 의존성을 줄이기 위한 효과적인 방법을 배울 수 있었습니다. 그 중 하나가 바로 `추상화 벽(Abstraction Wall)` 개념입니다.

**추상화 벽이란?**

`추상화 벽`이란 우리가 필요한 최소한의 인터페이스만 노출시키고, 그 외의 기능은 Wrapping 함수를 통해 감추는 방법입니다. 이렇게 하면 외부 라이브러리와의 의존성을 최소화할 수 있고, 유지보수성을 크게 향상시킬 수 있습니다.

<br />

### 추상화 벽 사용시, 장점

`추상화 벽`을 사용하면 여러 가지 장점이 있습니다.

1. 관리 포인트 감소: 외부 라이브러리와의 관리 포인트를 줄일 수 있습니다.
2. 유지보수성 향상: 외부 라이브러리의 변경 사항이 있을 때, 해당 추상화 벽 코드만 수정하면 자동으로 사용하는 코드도 변경 사항에 반영됩니다. 모든 코드를 수정할 필요가 없어 유지보수성이 높아집니다.
3. 복잡성 감소: 외부 라이브러리의 복잡한 내부 구조를 사용하는 코드에서 감출 수 있어 코드의 의존성을 최소화할 수 있습니다. 이로 인해 코드의 가독성이 높아지고, 변경도 유연하게 가능합니다.

<br />

### 추상화 벽 사용 예시

아래 코드는 카카오톡 공유하기 기능을 사용하기 위해 작성된 코드입니다.
이 코드에서는 추상화 벽을 세워 필요한 인터페이스만 노출시키고, 나머지 기능은 Wrapping 함수를 통해 가려놓았습니다.

```jsx
const shareKakao = ({ title, description, imageUrl, url, campaign }) => {
  if (typeof window.Kakao !== 'undefined') {
    window.Kakao.Share.sendDefault({
      objectType: 'feed',
      content: {
        title,
        description,
        imageUrl,
        link: {
          webUrl: url,
          mobileWebUrl: url,
        },
      },
    });
  }
};

export default shareKakao;
```

- 이 코드를 사용하면 카카오톡 공유 기능에서 문제가 생기거나 변경 사항이 있을 때, 이 코드만 수정하면 이를 사용하는 모든 코드에 자동으로 반영됩니다. 이렇게 하면 외부 라이브러리와의 의존성을 최소화하고, 유지보수성을 향상시킬 수 있습니다.

<br />

### 다른 추상화 벽의 예시

추상화 벽은 다양한 개발 도구와 프레임워크에서도 활용됩니다. 아래는 그 예시입니다.

1. React의 Virtual DOM

- React는 Virtual DOM을 사용하여 추상화 벽을 만들어 놓았습니다. 이를 통해 실제 DOM을 직접 조작하는 대신, Virtual DOM을 사용하여 UI를 효율적으로 업데이트할 수 있습니다.
- ex) `useState, setState`

2. Redux의 상태 관리

- Redux는 상태 관리 라이브러리로서 상태 변경을 위한 함수들만 노출시켜 추상화 벽을 제공합니다. 이로 인해 애플리케이션 전체에서 상태를 일관성 있게 관리할 수 있습니다.
- ex) `createStore, dispatch`

<br />

### 느낀점

이번 개발을 통해, 평소에 일상적으로 사용하던 여러 메서드들이 추상화 벽을 통해 관리되고 있음을 깨달았습니다. 앞으로 다른 사람의 코드를 볼 때, 작성 의도를 더욱 고려하고 프레임워크나 라이브러리 메서드를 사용할 때 그 목적과 개발 패턴을 명확히 파악하는 것이 중요하다는 생각이 들었습니다.
또한, 당연하게 받아들였던 많은 부분들이 이제는 당연하지 않게 느껴집니다. 이런 생각들이 앞으로 더 나은 코드를 작성하는 데 큰 도움이 될 것 같습니다.
